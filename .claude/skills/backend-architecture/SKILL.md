# Backend architecture — Go API для пользовательского состояния

Решение — [ADR 0002](../../../docs/adr/0002-backend-now.md): **гибрид**. Игровые данные остаются static-first (пайплайн → JSON на CDN); сервер (`server/`) держит **только изменяемое/общее состояние**: аккаунты, сейвы забегов, мета-прогрессия, лидерборд, дейлик-сид, Manager-персистенс.

## Слои (`server/internal/*`, не пробивать — как стадии пайплайна)
- **`transport/`** — HTTP: router (chi), хендлеры, DTO ↔ JSON, статусы/ошибки. Тонкий, без бизнес-логики.
- **`service/`** — бизнес-логика (забеги/лидерборд/сейвы). Не знает про HTTP и про SQL напрямую; принимает интерфейсы store.
- **`store/`** — доступ к Postgres (типизированные запросы), интерфейсы репозиториев. **Единственное место с SQL.**
- **`model/`** — доменные типы. Пересекающиеся с игрой — те же формы, что `schema/` (единый `accountId`).

## Правила
1. **Границы.** DTO не течёт в `service`; SQL не течёт в `transport`; `service` зависит от интерфейсов `store`, не от драйвера.
2. **Ошибки — единый контракт.** Один доменный тип ошибки + маппинг в HTTP (status + JSON `{code,message}`). Не плодить форматы по хендлерам.
3. **Секреты/конфиг — из env** (как в пайплайне), не в код; в логах редактировать токены.
4. **Миграции** версионированы, идемпотентны, в репо; схема БД — осознанная, не «как получилось».
5. **Контракт с фронтом.** Статику **не** проксируй через сервер — она с CDN через `DataSource`. Динамику фронт берёт отдельным API-клиентом. Пересекающиеся сущности отдавай в форме `schema/`.
6. **Детерминизм / анти-чит.** Дейлик/лидерборд валидируются сервером: счёт/паки воспроизводимы в Go — **переиспользуй `pipeline/internal/{model,rating}` и scoring-логику**, а не доверяй клиентскому результату слепо. `seed + dataset + версия ⇒ тот же итог`.
7. **Версионирование.** Учитывай `ratingModelVersion`/`schemaVersion`: сейв/лидерборд со старой версией помечается, не сравнивается с новой напрямую.
8. **Без k8s.** Один сервис + managed Postgres (Fly.io/Render). Инфра минимальна.

## Discovery перед кодом
- Есть ли уже похожий хендлер/репозиторий/pool? Не заводи второй.
- Тип уже в `model/` или `schema/`? Переиспользуй, не клонируй.
- Нужна ли вообще серверная фича, или это сингл (клиент + localStorage)? См. ADR 0002.

## Что НЕ делать
- Не класть игровые данные в БД (они статика).
- Не дублировать scoring копипастой — переиспользуй Go-пакеты пайплайна.
- Не хардкодить секреты; не смешивать слои; не заводить свой формат ошибок.
- Не поднимать k8s.
