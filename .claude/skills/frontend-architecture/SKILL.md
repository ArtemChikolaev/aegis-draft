# Frontend architecture — design system, UIkit, features

Слои `web/src`: `app/` (шелл + providers) · `design/` (tokens + base + theme) · `i18n/` (строки) · `ui/` (примитивы) · `features/` (экраны) · `game/ data/ state/ types/` (логика, не трогать из UI). Карта — [web/README.md](../../../web/README.md).

## Правило 0 — переиспользуй примитив, не создавай новый
Перед новым UI-элементом открой barrel **`web/src/ui/index.ts`**. Уже есть: `Button` (primary/primaryInvert/secondary/danger/leave/back), `Surface`, `Eyebrow`, `Banner`, `Chip`, `RoleTag`, `SoonBadge`, `StatTile`, `Select`, `Modal`, `OptionGroup`.

Формула: «**НЕ создаю кнопку/карточку/бейдж/модалку — беру `Button`/`Surface`/`RoleTag`/`Modal`; создаю новый примитив только если аналога нет**».
- Нужен ≥2 экранам → новый примитив в `ui/` (`Name.tsx` + `Name.module.css`), экспорт из `index.ts`.
- Одноразовая раскладка экрана → в `features/<screen>/<screen>.css` (глобальные классы раскладки), НЕ в `ui/`.
- Мало вариантов у существующего примитива? Добавь `variant`/`prop`, а не форкай новый компонент.

## Цвета — только токены (иначе темы ломаются)
- Любой цвет — переменная из `design/tokens.css`. **Ноль хардкод-цветов** (`#hex`, `rgba(...)`) в компонентах и tsx.
- Тема-зависимое → токен переопределяется в блоке `html[data-theme="light"]` **в tokens.css**. Всегда-тёмные панели/радар/модалки → инвертные токены (`--surface-invert`, `--on-invert`, `--muted-invert`, `--line-invert*`, `--brand-*`).
- Нужен новый цвет → **добавь токен**, а не литерал. Тогда light/dark работают сами.
- ❌ **Никогда не пиши `html[data-theme="light"] .foo { … }` в компонентном/feature CSS** — это ровно тот костыль, который мы вычистили (14 правил → 0). Он означает, что цвет не токенизирован. Токенизируй.
- Единственное исключение: декоративные брендовые градиенты (`mode-card`, `hero-art`) — фикс-арт, одинаковый в обеих темах, живёт в feature CSS.

## Адаптив — канон breakpoints (иначе wide/narrow разъезжаются)
- Ширины только из `design/breakpoints.css` / `breakpoints.ts`: **sm 430 / md 680 / lg 980**. В `@media` — литералы (`max-width: 680px`), не `var(--bp-*)` (MQ их не резолвит).
- Раскладка экрана — **explicit `grid-template-areas`** (или container queries у виджета), не «N детей текут в M колонок» без areas: на узком всплывут дубли/утечки текста.
- Новый `@media` с разовой шириной (900, 620…) — сначала сверь с каноном; не плоди четвёртый bp без правки `breakpoints.*`.
- **Один экран — один bp для слома колонок.** Реальная шишка: тим-панель ломалась на 980, а группы/отчёт того же экрана на 900 — в зазоре 901–980 страница разъезжалась сама с собой (панель уже узкая по центру, таблицы ещё в две колонки).
- Поведение (не только вид) тоже бывает адаптивным — тогда порог из JS: `isNarrowViewport()` в `design/breakpoints.ts`, а не свой `matchMedia` со своим числом. Пример: камера плей-оффа на широком ставится один раз на всю сетку, на узком ведёт за матчем юзера.

## Оверлеи (`ui/Modal`) — инварианты, которые легко потерять
- `aria-modal` без фокуса внутри — фикция: на открытии фокус уходит в диалог (`[autofocus]`, иначе крестик), Tab заперт внутри, на закрытии возвращается вызвавшему элементу.
- Фон под открытой модалкой **не скроллится** (`body overflow:hidden` + компенсация ширины скроллбара, иначе десктоп дёргается вбок).
- Отложенные таймеры выхода снимай на unmount: экран под модалкой может смениться раньше (reset забега).
- Скроллеры (списки внутри панелей) обрезай **по дуге рамки** — `border-radius` + `clip-path: inset(0 round …)`, а не распоркой-пустышкой в конце списка.

## Общие классы-словари (тир, состояние) — грабли
- **Общее правило не трогает `position`.** Хосты позиционированы по-разному: поставили `position: relative` в общий `.card-tint--*` — и карточки радара (`absolute` по вершинам) сорвало с мест. Контекст для `::after` обеспечивает поверхность, а не словарь.
- **Всегда-тёмная панель ремапит токены у себя**: `.on-invert-surface` переопределяет `--tier-*` на инвертный набор один раз на контейнере. Не заводи второй набор классов и не пиши `html[data-theme=…]` в компоненте.
- **Выразительность по размеру.** Крупная карточка — кант + движение; мелкая (≤80px, и их много сразу) — только кант: бегущий блик там читается как мигание, а виньетка на тёмном уходит ниже порога.
- **Звёздочка глоба вплотную к слэшу закрывает CSS-комментарий.** Перечисление классов через слэш в комментарии убивает идущие следом объявления, причём **сборка остаётся зелёной** — невалидные объявления просто отбрасываются. Наружу вылезает далеко: пустой токен ⇒ невалидный градиент ⇒ `background-image: none` ⇒ текст с `-webkit-text-fill-color: transparent` становится невидимым. Держит `web/test/tokens.test.ts` (проверяет postcss-AST, а не текст).

## Строки — только i18n
- Весь видимый текст — ключ в `i18n/core.ts` (типобезопасный `MessageKey`), RU и EN **синхронно**. **Ноль хардкод-строк** в разметке.
- Примитивы `ui/` презентационные: строку передаёт вызывающий (экран переводит через `t()`), сам примитив i18n не тянет. Так примитив остаётся переиспользуемым.

## Темы
- `ThemeProvider` (`design/theme`) ставит `data-theme` на `html`, хранит выбор, слушает систему. Компоненту НЕ надо знать тему — токены делают всё.
- Проверяй новый UI на **реальных скринах обеих тем** (light + dark) и на **полном flow** (start → draft → result), а не на изолированном компоненте. Продуктовые/дизайн-правки — тоже на скринах, не «на глаз».

## Визуальная айдентика
Характер, палитра dark/light и тон — в [docs/design-language.md](../../../docs/design-language.md) (design-spec). Кратко: Dota tournament roguelike (не B2B/курсы); dark = pure black + редкий green art; light = ivory + orange + black inserts; цвет = арт/сигнал, не заливка. 322-0 — только UX-референс, не pixel-copy.

## Инструкции
1. `ui/index.ts` → найди подходящий примитив, переиспользуй (или добавь variant).
2. Цвет? → токен из `design/tokens.css` (или добавь новый токен, оба блока при тема-зависимости).
3. Текст? → ключ в `i18n/core.ts` (RU + EN).
4. Экран? → папка `features/<screen>/`, собери из `ui/`, раскладку в локальный `<screen>.css` с explicit grid-areas и `@media` только на bp sm/md/lg.
5. Прогони `tsc --noEmit` + `vite build`; глянь light + dark и **wide + narrow** вживую.

## Что НЕ делать
- Не лепить bespoke `<div className="…">` со стилями, если есть примитив в `ui/`.
- Не хардкодить цвет или строку.
- Не писать `html[data-theme=…]`-override в компонентах.
- Не тянуть i18n/тему внутрь `ui/`-примитива (он презентационный).
- Не размещать раскладку экрана в `ui/`, а примитив — в `features/`.
