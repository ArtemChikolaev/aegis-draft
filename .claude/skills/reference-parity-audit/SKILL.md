# Reference parity audit

Сравнивать поведение, а не названия компонентов. Цель — обнаружить дефекты и продуктовые пробелы, не сделать слепую копию.

## Обязательный процесс

1. **Зафиксировать рамку.** Назвать референс, проверяемый сценарий, дату, версию/commit нашей реализации, конфиг и seed. Если scope не задан — пройти golden path от старта до результата и по одному важному edge case на изменённую механику.
2. **Поднять факты о проекте.** Прочитать `CLAUDE.md`, `docs/PRD.md`, `docs/BACKLOG.md`, релевантный код и тесты. Не считать статус ✅ в backlog доказательством работы.
3. **Пройти референс живьём.** Для интерактивного продукта использовать браузер, записать доступные состояния, правила, обратную связь, ошибки и переходы. Для Balatro брать продуктовые принципы (эскалация, экономика, build-синергии, боссы, мета-прогресс), а не копировать контент, визуал или формулы.
4. **Пройти нашу реализацию тем же сценарием.** Проверить UI и чистую логику; запустить релевантные тесты. Сравнивать одинаковые настройки и состояния, где это возможно.
5. **Заполнить матрицу из `references/audit-template.md`.** Каждую строку пометить одним статусом: `parity`, `intentional-divergence`, `defect`, `missing`, `not-applicable`, `unknown`.
6. **Проверить подозрения.** Для `defect` дать воспроизводимые шаги и сослаться на код/тест. `unknown` не превращать в задачу без дополнительной проверки.
7. **Исправить в пределах запроса.** Небольшой подтверждённый дефект исправить и перепроверить. Большую или продуктово неоднозначную работу оформить атомарной задачей в backlog, не расширяя scope молча.
8. **Синхронизировать документы:**
   - новое/изменённое продуктовое требование → `docs/PRD.md`;
   - подтверждённый implementation gap/bug → `docs/BACKLOG.md` с файлами, DoD и зависимостями;
   - повторяемая процессная шишка → этот скилл или `CLAUDE.md`;
   - разовый факт аудита → датированный файл `docs/audits/YYYY-MM-DD-<topic>.md`.
9. **Проверить отчёт.** Если создан `docs/audits/*.md`, запустить `node .claude/skills/reference-parity-audit/tools/validate_audit.mjs <path>`.
10. **Перепроверить.** Повторить затронутый сценарий, тесты и сборку. В финале отделить исправленное сейчас от запланированного.

## Правила качества

- Не сравнивать только скриншоты: проверять состояния, ограничения, расчёты, ошибки и завершение сценария.
- Не записывать вкусовое отличие как баг. Объяснять, какую проблему пользователя решает заимствование.
- Не ставить ✅ по ручному впечатлению, если DoD требует тест/сборку/данные.
- Не удалять осознанные отличия Aegis Draft: единый `accountId`, сглаживание, прозрачная модель рейтинга, Mixed Draft.
- Не утверждать parity, если референс недоступен: поставить `unknown` и указать ограничение.
- Не оставлять PRD и backlog противоречащими коду после подтверждённого аудита.

## Минимальный результат

- матрица с доказательствами;
- список P0/P1/P2 gaps;
- воспроизведение каждого дефекта;
- обновлённые PRD/backlog там, где изменились продуктовые требования или план;
- результаты повторных тестов.
