# Plan-first communication — план до кода, вопрос до догадки

Цена «2 минуты на согласование» меньше цены «час на переделку».

## Когда применять
- Задача затрагивает 2+ слоя (pipeline / web / schema).
- Появляется новая сущность (Go-пакет, TS-модуль, файл схемы, стадия ETL).
- Задача расходится с PRD (`docs/PRD.md`) или ADR — подними флаг, не решай молча.
- Есть два одинаково разумных подхода — спроси, не угадывай.
- Что-то заскочило в scope, но не было в задаче.

## Не используй для
- Опечатка, одна строка, очевидный фикс в одном файле, переименование.
- Когда пользователь явно сказал «делай как считаешь нужным».

## Шаблон плана (5–10 пунктов)
1. **Что меняем по делу** — что должно работать после задачи.
2. **Какие файлы трогаем** — список путей.
3. **Новые сущности** — Go-пакеты/TS-модули/файлы схемы.
4. **Что переиспользуем** — конкретные ссылки (см. [[discovery-before-code]]).
5. **Контракт данных** — меняется ли `schema/`? Нужен ли ре-ген TS-типов / бамп `manifest`? (см. [[data-contract]]).
6. **Рейтинг** — трогаем ли формулы? Нужен ли бамп `ratingModelVersion`? (см. [[scoring-model]]).
7. **Open questions / риски** — что неясно, edge cases.
8. **Что НЕ делаем в этом заходе.**

Вопрос в конце: **«Ок? Идём по этому плану?»**

## Правила общения
- **Амбигвальность → спроси, не угадывай.** ✅ «Прежде чем сделать X (можно ещё Y), какой вариант?» вместо ❌ «Я сделал X, потому что…».
- **Противоречие с PRD/ADR → флаг.** Зафиксируй расхождение и спроси, а не маскируй.
- **Scope-creep → отдели.** «По пути увидел кривой X — вынести в отдельную задачу?»
- Для решений, которые реально за пользователем (выбор механики/подхода) — используй вопрос, а не додумывание. Открытые вопросы PRD §10 — типичные кандидаты.
