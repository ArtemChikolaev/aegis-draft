# Scoring model — зафиксированные правила рейтингов

Модель рейтингов версионируется (`manifest.ratingModelVersion`). **Любое изменение формулы/весов/окна ⇒ бампни версию.** Решения ниже уже согласованы в PRD §5 — не переизобретай их.

## Зафиксированные решения (не менять без согласования)
1. **Роли — как в 322-0, без деления 4/5.** `role ∈ {safelane, mid, offlane, support}`, support ×2, слоты взаимозаменяемы. Причина: приоритет фарма ненадёжен (5-ка фармит, 4-ка варды). PRD §5.1.
2. **`Team OVR = Base + Hero Synergy + Chemistry`.**
3. **Base:**
   - **Event Rating** — форма игрока на событии пака (Team Packs).
   - **Peak Rating** — career-best по **скользящему окну 3–6 мес.** (НЕ лучший одиночный турнир), порог `games ≥ N_min`. PRD §5.4.2.
   - **Team-Success Rating** (Mixed Draft) — по успеху команды за окно (1/2/5 лет): placements/призовые/винрейт на тир-1, поправка на инд. форму. PRD §5.4.3.
4. **Hero Synergy — сглаживание обязательно:** `score = (winrate·games + m·μ) / (games + m)`, μ≈0.5, m≈10. Сырой winrate при малых `games` НЕ использовать. Назначение 5 героев 5 игрокам — **задача о назначениях** (венгерский алгоритм / max-weight matching), а не жадность.
5. **Chemistry** — сглаженный бонус за сыгранность пар из `squadSynergy`; в Mixed обычно ниже (осознанный трейд-офф).
6. **OVR/IMP/ECO/REL** нормируются в 0–100, веса свёртки **зависят от роли** (кэрри ≠ саппорт).
7. **Mixed pack** — ровно 5 ролевых слотов, все кандидаты из разных команд; порядок выбора свободный среди незаполненных ролей. После пика роль блокируется, support доступен до заполнения обоих взаимозаменяемых support-слотов. Если валидный lineup собрать нельзя, fail-fast; не повторять команду и не сдвигать индексы fallback-ом.

## Правила реализации
- **Где живёт логика:** детерминированные агрегаты и Base/Peak/team-success считает **Go-пайплайн** (`internal/rating`), пишет числа в data JSON. **Сглаживание synergy, назначение героев и итоговый Team OVR** считает **TS-фронт** (`web/src/game/*`), чтобы модель сглаживания менялась без пересборки данных (см. [[data-contract]] инвариант 4).
- **Параметры окна/сглаживания** (`m`, μ, длина окна, `N_min`, веса ролей, веса престижа TI/Major) — в одном месте-конфиге, не размазаны по коду. Версионируй вместе с моделью.
- **Малые выборки** — всегда сглаживай/ставь порог; одна игра 100% winrate не должна давать пик или синергию.
- **Когорта рейтинга явная** — role-relative percentile/normalization считается отдельно для одного event/window scope; API расчёта обязан требовать scope, чтобы события и окна не смешались молча.
- **Роль приходит из normalize/roster** — rating-слой не угадывает carry/mid/offlane/support по GPM или одной карте. Оба support-слота используют одну формулу без скрытого деления 4/5.
- **Peak v1.1.0** — 120 календарных дней включительно, `games ≥ 15`, отдельно по ролям. Считать в change-points входа/выхода матчей (эквивалент ежедневного rolling window); tie-break: OVR → games → более раннее окно. Одиночный турнир ниже порога peak не создаёт.
- **Team-Success v1.2.0** — отдельные nested окна 1/2/5 лет; TI/Major/tier-1 prestige = 3/2/1; компоненты Placement/Prize/smoothed Winrate/Top = 40/20/25/15. Хранить сырой `games+winrate`. Несколько команд игрока взвешивать по играм; individual OVR — только bounded multiplier 0.8–1.2. Неизвестный teamId fail-fast; `valve_legacy` не подменять обычным 5y.

## Чек перед «готово» (рейтинги)
- [ ] Менял формулу/веса/окно → `ratingModelVersion` бампнут.
- [ ] Малые выборки сглажены (нет сырого 0/1 winrate в счёте).
- [ ] Назначение героев — matching, не жадность.
- [ ] Роли не поделены на 4/5.
- [ ] Параметры в конфиге, не хардкодом по файлам.
