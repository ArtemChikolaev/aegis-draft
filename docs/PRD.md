## 1. Краткое резюме (TL;DR)

Roguelite про **драфт «команды всех времён»**. Игроку выдаются «паки», из каждого он забирает игрока в состав по ролям, раздаёт героев и максимизирует `Team OVR = Base + Hero Synergy + Chemistry`. Цель — собрать сильнейшую пятёрку и «пройти International всухую».

**Ключевой факт про оригинал:** 322-0 — это **статичный SPA** (Vite), грузящий 7 предрассчитанных JSON из `/data/*.json`. Рантайм-бэкенда нет. Вся сложность (и все баги — «странный парсинг») сидят в **офлайн-пайплайне** генерации этих данных.

**Наша ставка:** главная инженерная ценность — прозрачный, версионированный **data-пайплайн на Go**; фронт на **TS/React**; данные отдаём **статикой** (static-first), сервер добавляем позже под мета-фичи.

## 2. Разбор оригинала (322-0)

Название — отсылка к мему «322» (договорняк) и к идее «flawless» (безупречный проход турнира).

### 2.1. Настройки старта (оригинал)
| Параметр | Опции оригинала |
|---|---|
| Режим | `322-0 Classic`, `Esports Manager (beta)` |
| Game Mode | `Classic` (5 игроков + 5 героев в паке) |
| Format | `Valve Legacy`, `Standard (~2 года)` |
| Difficulty | `Hard` (0), `Easy` (1), `Smurfing` (2 реролла) |
| Scoring | `Event Rating`, `Peak Rating` (soon) |
| Hero Allocation | `Automatic`, `Manual` |

### 2.2. Игровой цикл (Classic)
1. Показывается **пак** = одна команда на одном турнире → 5 игроков (роли + рейтинги) + 5 сигнатурных героев.
2. Забираешь **одного игрока** → он встаёт в слот своей роли на пентагоне.
3. Появляется следующий пак. Повтор, пока не заполнены 5 ролей.
4. Параллельно копятся герои → распределяются по игрокам (авто/вручную).
5. **Reroll** заменяет пак (лимит = сложность).
6. Итог: `Team OVR` = `Base + Hero Synergy + Chemistry` + радар-пентагон.

### 2.3. Формула
```
Team OVR = Base + Hero Synergy + Chemistry
```
- **Base** — средний event-OVR пятёрки.
- **Hero Synergy** — бонус за оптимальное назначение героев (по historical player×hero: игры × винрейт).
- **Chemistry** — бонус за сыгранность выбранных игроков вместе.

3. **Непрозрачный OVR.** → Явно **специфицируем и версионируем** модель (§5).

Ядро статистики — **OpenDota** (по `hero_id` = Valve constants и account_id видно, что это не Dotabuff). Метаданные турниров — **Liquipedia**.

| Данные | Источник | Статус / примечание |
|---|---|---|
| Турниры, даты, патчи, команды, ростеры, placement, лого | **Liquipedia API** | ⛔ Нужен одобренный API access. На 2026-07-11 Basic временно недоступен; free-доступ возможен по заявке для подходящих open-source/non-commercial проектов. Реализация следует выданной OpenAPI-спеке/лимиту; обязательны выданная auth scheme, контактный User-Agent, кэш и атрибуция CC-BY-SA. Wiki/MediaWiki scraping как обход не используем. |
| Pro-матчи, пики/бэки, результаты | **OpenDota API** | ✅ Free Tier без ключа: текущая конфигурация сервиса — 60 req/min и 3000 req/day. Premium key отдельный от Steam API key, требует OpenDota login/billing и даёт более высокий лимит. |
| Player×hero игры/винрейты | **OpenDota** (`/players/{id}/heroes`, агрегация матчей) | Основа Hero Synergy / Peak. |
| Сыгранность пар, тиммейты (Chemistry) | **OpenDota** (`/players/{id}/peers` + агрегация pro-матчей) | `/peers` даёт пожизненные совместные игры с каждым тиммейтом — прямой источник Chemistry. |
| Картинки героев/лого | Valve CDN / Liquipedia | Кэшируем локально. |

### 4.1. Про Dotabuff / Datdota (ответ на вопрос «можно ли парсить и не проще ли брать всё с Datdota?»)
- **Liquipedia — да**, только через одобренный официальный API access (см. выше). До одобрения источник заблокирован, а не заменяется скрейпингом.
- **Dotabuff — технически можно, но не рекомендуется.** У Dotabuff **нет публичного API** — только скрейпинг HTML, что: (а) противоречит их ToS и защищено Cloudflare; (б) хрупко (вёрстка меняется); (в) **избыточно** — всё, что показывает Dotabuff, доступно через OpenDota/STRATZ API легально.
- **Datdota — не источник для нас (решение 2026-07-12).** 322-0 внизу пишет «stats from Datdota», но это не пример для копирования: **Datdota и 322-0 — один автор (Noxville)**, он ходит в собственную БД, а не скрейпит. У Datdota **нет публичного API**, данные проприетарны, скрейп против ToS — тот же класс, что Dotabuff. Воспроизвести путь 322-0 мы легально не можем, и это не нужно: всё, что нам не хватает (пожизненные co-games, career/event player×hero, tier-1 фильтр по лигам), **OpenDota отдаёт напрямую** (`/players/{id}/peers`, `/players/{id}/heroes`, `/proMatches`+`/matches/{id}`+`/leagues.tier`). Разрыв с 322-0 был не в источнике, а в объёме собранного (один турнир вместо мульти-эвентного окна).
- **Вывод:** **OpenDota (+ Liquipedia для метаданных) полностью покрывают потребность**; Dotabuff/Datdota — не основа и не дополнение.

> Принцип: сырьё тянем один раз в raw-слой, детерминированно пересчитываем в игровые JSON. Пользователь в рантайме к внешним API не ходит.

## 5. Игровые механики (что специфицируем сами)

### 5.1. Роли — как в 322-0, без деления саппортов
Деление на semi/hard support **отклонено**: по игре часто 5-ка уходит фармить, а 4-ка качает варды — приоритет фарма ненадёжно отражает позицию, будет много ошибок классификации. Оставляем схему оригинала:

| Слот | Роль | `role` |
|---|---|---|
| 1 | Safelane / Carry | `safelane` |
| 2 | Mid | `mid` |
| 3 | Offlane | `offlane` |
| 4 | Support | `support` |
| 5 | Support | `support` |

`role ∈ {safelane, mid, offlane, support}`, где `support` встречается дважды. Два саппорт-слота **взаимозаменяемы** (любой саппорт ложится в любой из двух).
Пак обязан содержать валидную пятёрку ролей, но может включать substitutes сверх пяти; схема и UI не должны отбрасывать их молча.
OpenDota `lane_role` не используется для искусственного деления support на position 4/5: оба игрока остаются просто `support`, как в 322-0. Для M2 player×hero/chemistry роли не требуются вообще.

### 5.2. Рейтинг игрока (Base / OVR)
Из матчей игрока агрегируем и нормируем в 0–100 **с весами по роли**:
- **IMP (Impact)** — KDA, участие в убийствах, урон по героям, teamfight-вклад.
- **ECO (Economy)** — GPM/XPM, net worth к таймингам, ластхиты (с поправкой на роль).
- **REL (Reliability)** — смерти, выживаемость, консистентность по картам.
- **OVR** — взвешенная свёртка (веса зависят от роли: кэрри ≠ support; деления support на 4/5 нет).
- Версионируется: `ratingModelVersion` в артефакте данных.

#### Rating model `v1.0.0`

Первая воспроизводимая модель использует только сигналы, реально доступные в текущих OpenDota match details:

| Компонент | Метрики | Веса |
|---|---|---:|
| IMP | `(K+A)/max(1,D)` · kill participation · hero damage/min | 35% · 30% · 35% |
| ECO | GPM · XPM · last hits/min | 45% · 35% · 20% |
| REL | deaths/10 min (меньше лучше) · консистентность KDA/GPM/damage между картами | 65% · 35% |

Каждая метрика сравнивается **внутри своей роли**, а не между carry и support. Сначала наблюдение сглаживается к среднему ролевой когорты: `x' = (n·x + k·μ_role)/(n+k)`, затем percentile-score дополнительно сжимается к нейтральным 50: `score = 50 + (percentile-50)·n/(n+k)`. В `v1.0.0` `k=8`; для consistency эффективная выборка равна `max(0, games-1)`. Так одна выдающаяся карта не превращается в рейтинг 100.

| Роль | IMP | ECO | REL |
|---|---:|---:|---:|
| Safelane | 40% | 45% | 15% |
| Mid | 45% | 40% | 15% |
| Offlane | 45% | 25% | 30% |
| Support ×2 | 45% | 15% | 40% |

Оба support-слота используют одну формулу. Net-worth timings и полноценный teamfight contribution не подменяются догадками: они могут войти в следующую версию только после появления надёжных данных. Назначение роли приходит из normalize/roster слоя; rating-модель сама роль не угадывает.

### 5.3. Режимы драфта: Team Packs vs Mixed Draft 🔄
Новая ось настройки — **Draft Style**:

- **Team Packs (Classic).** Пак = реальный ростер **одной команды** на турнире (5 игроков всех ролей); берёшь одного игрока (как в оригинале). Скоринг — event/peak рейтинг игрока.

- **Mixed Draft.** 🔄 Пак = **5 игроков из РАЗНЫХ команд**, по одному на роль (напр. `Ame` (carry) · `bzm` (mid) · `33` (off) · `Boxi` (sup) · `Kataomi'` (sup) — каждый из своей команды). То есть внутри одного ролла — не ростер команды, а синтетический «микс-лайнап» из звёзд разных команд. Берёшь **одного** игрока в свой состав, остальные — контекст/превью; реролл даёт новый микс-лайнап.
  - **Порядок ролей — строгий `1→5`** (сначала carry, потом mid, off, sup, sup): из каждого пака драфтишь кандидата под **текущую** заполняемую роль (или реролл, если кандидат не нравится). *(Открытый нюанс §10-A: разрешить ли брать любую роль из микса — «свободный порядок». Дефолт — строгий 1→5.)*
  - **Скоринг игроков в Mixed — по успеху команды за окно** (не по форме на конкретном событии): рейтинг игрока = функция от результатов его команды(-д) за выбранный период (**1 / 2 / 5 лет**, та же ось Format, что и в классике). Успех = placements/призовые/винрейт на тир-1 ивентах за окно, с поправкой на индивидуальную форму. Так «Ame из чемпионского состава» весит больше, чем тот же игрок из команды-аутсайдера. Детали формулы — §5.4.3.
  - **Chemistry обычно ниже** (звёзды из разных команд не сыграны вместе) — осознанный трейд-офф «сильные индивидуально vs сыгранные». Источник реиграбельности.
  - **Герои:** каждый игрок-кандидат приносит свои сигнатурные герои; при пике добавляем их в общий пул.

> Оси независимы: `Draft Style × Format × Difficulty × Scoring × Allocation`.
> Как валидировать очки игроков в Mixed — см. §5.4.3 (team-success rating).

### 5.4. Scoring
#### 5.4.1. Event Rating
Форма игрока **на событии пака**. Один и тот же игрок на разных турнирах = разный счёт (это фича). Используется в Team Packs.
- **Per-event, а не career:** Base игрока = как он **и его команда суммарно отыграли конкретный турнир**. Один и тот же игрок может быть 60 на одном ивенте и 90 на другом. Это и есть основной baseline (career/peak — отдельная ось Scoring, §5.4.2).
- **Scope = только tier-1 сцена** (подтверждённое решение 2026-07-12). Правило tier-1 = **exclude-based + порог матчей**, потому что OpenDota-тир ненадёжен: `premium` — всего ~214 лиг и не покрывает реальные топ-турниры (EWC 2026, DreamLeague S19/23/24, OGA PIT, EPICENTER помечены `professional`). Поэтому:
  - **tier-1 = `premium` ∪ (`professional` − шум)**, где шум — квалы/дивизионы/регионалки/минорки/бегиннеры и т.п. (паттерны в `pipeline/internal/tier1`);
  - **порог**: события с `< N` матчей в окне отбрасываются (`--min-event-matches`, дефолт 8) — гасит мелкий остаток шума и недосбор;
  - `valve_legacy` = все The International (по имени) + курируемый набор Valve/DPC Major id (классические Kiev/Boston/Shanghai в OpenDota — `professional`, по тиру не выделить).
  - Не-tier-1 матчи в рейтинг/синергию/химию не входят.

#### 5.4.2. Peak Rating — скользящее окно 🔄
Career-best форма игрока, **не привязанная к турниру пака**. Определение:
- Peak = **максимум по карьере** рейтинга игрока в роли драфта, посчитанного по **скользящему окну 3–6 месяцев** (а не по лучшему одиночному турниру).
- **Почему окно, а не турнир:** иначе игрок, сыгравший один турнир 100-0, а все остальные проваливший (не выходя из группы), получил бы незаслуженно высокий peak. Окно требует устойчиво высокой формы на протяжении месяцев.
- **Порог выборки:** окно засчитывается только при `games ≥ N_min`, иначе не считается пиком.
- В `ratingModelVersion=v1.1.0`: окно — **120 календарных дней включительно**, `N_min=15`. Пересчёт выполняется во все даты, когда матч входит в окно или выходит из него; это эквивалентно ежедневному rolling-проходу без пустых повторных вычислений.
- Peak хранится отдельно по каждой роли. При одинаковом OVR выбирается окно с большим числом игр, затем более раннее окно — результат не зависит от порядка входных данных.
- Параметры окна и `N_min` версионируются; изменение требует новой `ratingModelVersion`.

#### 5.4.3. Team-Success Rating (для Mixed Draft) 🆕
В режиме **Mixed Draft** игрок оценивается **по успеху его команды за окно** (1/2/5 лет — ось Format), а не по индивидуальной форме на событии:
- Базовые сигналы за окно: **placements** на тир-1 ивентах (веса по престижу: TI > Major > прочее), **призовые**, **винрейт** команды, кол-во топ-финишей.
- Итог: `PlayerScore = f(TeamSuccess за окно) · g(индивидуальная форма)` — команда-успех как основной множитель, инд. форма как поправка (чтобы «пассажир» чемпионского состава не весил как его кор).
- Если игрок за окно поиграл в нескольких командах — агрегируем по времени, проведённому в каждой (weighted by games/сроки).
- Формула и веса версионируются, тюнинг на данных (§10-C).

В `ratingModelVersion=v1.2.0` для каждого `last_1y/last_2y/last_5y`:

- `PlacementScore = clamp(25 · Σ prestige(event) · placementValue)`;
- prestige: **TI=3, Major=2, tier-1=1**;
- placement value: **1-е=1.00, 2-е=0.65, top-4=0.40, top-8=0.15**;
- `PrizeScore = clamp(100 · log1p(prizeUsd) / log1p(10 000 000))`;
- winrate для формулы сглаживается как `(wins + 10·0.5)/(games+10)`, а в `teamSuccess.json` сохраняются сырой `winrate` и `games`;
- `TopScore = clamp(titles·30 + top4·10)`;
- `TeamSuccess = 40% Placement + 20% Prize + 25% Winrate + 15% TopScore`.

Для игрока: сначала успех его команд усредняется по числу сыгранных за них матчей, затем умножается на ограниченную поправку индивидуальной формы: `factor = 0.8 + 0.4·IndividualOVR/100` (диапазон `0.8–1.2`, OVR 50 нейтрален). Итог ограничивается `0–100`. Неизвестный `teamId` — ошибка контракта, а не нейтральный fallback. `valve_legacy` требует отдельного курируемого набора TI/Valve Major и в эту временную агрегацию намеренно не входит.

| Опция | Пул |

| `last_2y` | За последние **2 года** (бывш. «Standard») |

| `Normal` | 1 |

### 5.7. Hero Synergy (оптимальное назначение) — без изменений по сути
- Сглаженный скор пары (игрок, герой): `score = (winrate·games + m·μ)/(games + m)`, μ≈0.5, m≈10.
- Назначение 5 героев 5 игрокам с максимизацией суммы — **задача о назначениях** (венгерский алгоритм). Manual — игрок сам, UI подсказывает оптимум.
- **Приоритет матчинга — число игр** (`assignmentPairScore = games·1000 + winrate`): «больше игр — лучше», как в 322-0 (наблюдалось: Void Spirit переехал с игрока с 7 играми на игрока с 30). Automatic ре-оптимизирует всё назначение при каждом новом игроке; остаточному игроку может достаться герой с 0 игр. Роль героя назначение **не** ограничивает (у референса так же).
- **Источник player×hero — career + event** (`/players/{id}/heroes` + агрегация матчей события), только tier-1. Глубина этих данных прямо определяет силу синергии; тонкий срез одного турнира её занижает.
- Во время незавершённого драфта среднее считается только по уже назначенным player×hero-парам; игрок без выбранного героя не создаёт искусственный штраф. Каждый новый пак всегда предлагает ровно 5 ещё не взятых героев: при пересечении с уже выбранными сигнатурный набор детерминированно добирается из format-pool.
- UI показывает `games` назначенной пары. Клик по игроку в пентагоне открывает event-scoped и career player×hero `{games, winrate}` и внешний профиль `https://datdota.com/players/{accountId}`.

### 5.8. Chemistry — пожизненные попарные co-games 🔄
- **Основной сигнал — пожизненная сыгранность любых двух игроков состава**, а не только пар из одного пака. Подтверждено на 322-0: `Saksa + Watson · 185 games · +0.8` сработало для игроков, взятых из **разных** паков (Digital Chaos и Cloud9) — связь по совместной карьере на tier-1. UI показывает попарно `Игрок A + Игрок B · N games · +бонус`.
- Из `squadSynergy` (пары) — сглаженный бонус за сыгранность внутри итоговой пятёрки. Учитываем career (lifetime) + события паков; текущий состав (`teamId+eventId`) — полный вес, бывшие тиммейты — пониженный (`×0.35`).
- **Требование к данным:** сигнал живой только на **мульти-эвентном tier-1 датасете** с пожизненной матрицей co-games (`/players/{id}/peers` + агрегация pro-матчей). На срезе одного турнира кросс-командных пар нет → химия структурно ≈ 0 (текущий deploy). Это главный алгоритмический разрыв с 322-0, закрывается пайплайном (BACKLOG `TREF6`/`TDATA1`).
- Промежуточная Chemistry — накопительная проекция относительно 10 возможных пар полной пятёрки: новые сыгранные пары постепенно добавляют вклад. Для итоговых пяти игроков масштаб совпадает с формулой полной пятёрки.
- Игровой клиент не запрашивает OpenDota/Liquipedia во время драфта: справочник героев, player×hero и пары заранее собирает resumable Go ETL и публикует как versioned `public/data/*.json`. Runtime-БД для этих данных не нужна; расширение last_2y и scheduled refresh остаются задачей пайплайна.

### 5.9. Полный Roguelite Run 🟨
Первый исполняемый вертикальный цикл Classic реализован: после 5 пиков игроков + 5 пиков героев пользователь получает прогноз против поля из 18 команд, проходит две группы по 9, double-elimination плей-офф, Grand Final BO5 и итоговое место. Поле и все матчи воспроизводимы по `seed + dataset version`.

> **UI-статус 2026-07-12 (правки в рабочем дереве, деплой пользователем):** соперники Classic — **фэнтезийные боты** со случайными именами (реальные команды — режим Real Tournament); итог драфта ведёт в турнир кнопкой «Start tournament» (без «New run»/preview/seed); стадии `field → groups → playoffs`, где playoffs — **терминальный экран** (место + чемпион + сетка UB/LB с Grand Final + итоговая таблица + твой состав/герои/синергия/химия) — всё на одном экране, как в 322-0. **Осталось (P0): live-симуляция как процесс** — group stage «LIVE» + Skip, пораундовое раскрытие сетки/таблицы (сейчас результат считается мгновенно, экран лишь раскрывает готовое). См. BACKLOG `TREF-TOUR1`.

`TournamentEngine` — отдельный чистый orchestration-слой поверх результата драфта: он не расширяет `RunEngine` и не меняет формулу `Team OVR`. Группа — круговой BO2 (16 карт на команду); топ-4 идут в upper bracket, 5–8 — в lower, девятая команда вылетает. Итоговые диапазоны соответствуют точке вылета из double-elimination сетки. Прогноз ранжирует сырой OVR, а матч использует логистическую вероятность, поэтому фаворит может проиграть, но seed всегда даёт тот же результат.

Оставшаяся roguelite-арка — экономика, награды, Camp/Market, build-defining решения и boss conditions между турнирными этапами:

Полный забег строится как Dota-турнир из последовательных этапов:

1. **Draft** — собрать стартовый ростер и пул героев.
2. **Stage** — пройти реальную турнирную структуру (Group Stage → double-elimination Playoffs → Grand Final).
3. **Reward** — получить валюту и выбрать одно улучшение/перестройку.
4. **Market / Camp** — купить тактику, расширить hero pool, заменить игрока или подготовиться к следующему условию.
5. **Boss Stage** — особое правило, меняющее оптимальную сборку, а не просто больший числовой порог.
6. **Win / Loss** — не прошёл порог за доступные попытки — забег завершён; прошёл Grand Final — победа и итоговая карточка.

Сложность поля и специальных условий должна расти быстрее базовой силы состава, поэтому игрок обязан строить синергии, а не просто выбирать максимальный OVR. Seed определяет паки, поле, матчи, будущие рынок и награды целиком.

#### 5.9.1. Режимы забега

1. **Classic.** All-time fantasy draft; текущая механика. Внутри выбирается `Team Packs` или `Mixed Draft`, затем состав проходит турнирную арку. Это базовый и самый прозрачный режим.
2. **Esports Manager.** Игрок принимает организацию в выбранном регионе, работает с контрактами/бюджетом, собирает доступный ростер и идёт через квалификации и сезоны. В отличие от Classic, ценность игрока определяется не только OVR, но и стоимостью, сроком контракта, регионом и развитием. Точные правила — отдельная продуктовая спека ([docs/modes-scenarios.md](modes-scenarios.md), BA-черновик 2026-07-12); режим не должен быть просто Classic с другой обложкой.
3. **Real Tournament.** Seed выбирает реальный tournament snapshot и заранее показывает поле соперников (ориентир — 16–20 команд). Их заявленные на snapshot игроки получают **roster lock** и исключаются из пользовательского draft pool. Пользователь собирает challenger-команду из игроков вне турнира: исторических легенд, свободных агентов и игроков других эпох/сцен. Затем играет против фиксированного поля через группы, плей-офф и финал. Развёрнутый сценарий и переиспользование (`TournamentEngine` с реальными `packs` как opponentPool) — [docs/modes-scenarios.md §2](modes-scenarios.md).

Правила Real Tournament:
- eligibility считается по каноническому `accountId`, а не по нику; один человек не может одновременно быть у соперника и пользователя;
- snapshot включает event id, дату отсечки, команды и их заявленные ростеры; поздняя замена/stand-in учитывается только если входит в выбранный snapshot;
- opponent field, draft pool, рынок и матчи полностью воспроизводимы по seed + dataset version;
- исторический игрок оценивается своей формой выбранной эпохи, а не сегодняшним состоянием;
- если после roster lock нельзя собрать валидную пятёрку ролей, генератор обязан сменить snapshot/pool или завершиться понятной ошибкой, но не ослаблять lock молча.

### 5.10. Build-системы и мета-прогрессия 🆕
Применяем принципы Balatro, но переводим их на язык Dota:

- **Tactics** — ограниченный набор пассивных модификаторов забега с условиями (`+` за общий регион/эру, широкий hero pool, сыгранные пары, comeback-состав и т.п.). Это источник разных билдов; конкретные эффекты требуют отдельного balance spec.
- **Camp actions** — одноразовые действия между этапами: scrim, bootcamp, scouting, hero practice, stand-in/transfer. Они меняют решение сейчас, а не дают вечный бонус аккаунту.
- **Boss conditions** — ограничения вроде урезанного hero pool, штрафа несбалансированному ростеру, давления конкретной меты или запрета одного типа синергии. Условие показывается заранее, чтобы игрок мог адаптироваться.
- **Economy** — валюта за прохождение и overperformance; выбор между немедленным усилением и накоплением. Реролл рынка и реролл draft-пака — разные расходы.
- **Meta progression** — открывает новые tactics, форматы, stakes и challenge-наборы. Постоянные `+OVR` не выдаём: они размывают честность seed/challenge.
- **Replayability** — seeded run, daily challenge, escalating stakes, статистика открытий и история забегов.

Первая версия полного цикла должна доказать арку на 4 этапах с малым курируемым набором tactics; масштабировать контент только после телеметрии и симуляции баланса.

## 7. Технологический стек

### 7.1. Go — data-пайплайн (ядро ценности) + опц. API
ETL: тянуть OpenDota/Liquipedia (goroutines + worker pool + rate-limit), агрегировать матчи, считать рейтинги, генерить `*.json`. Прозрачно и тестируемо заменяет «странный парсинг» оригинала.

### 7.2. TypeScript — фронт + игровая логика
React + Vite + TS. State — Zustand. Пентагон — SVG/Canvas. Логика счёта на клиенте (как в оригинале) — быстро, без сервера.

### 7.3. Runtime: static-first сейчас, Go API позже — обоснование
Твой вопрос: «если для масштабирования лучше Go API — то его». Честный ответ: **для масштабирования сам static-first лучше и дешевле** — JSON на CDN масштабируется практически бесконечно и почти бесплатно, игровая логика на клиенте. Go API нужен **не для масштаба, а для фич с общим состоянием**: глобальный лидерборд, дейлик-сид с валидацией, сохранение забегов, анти-чит.

**Решение:**
- **Сейчас (static-first):** фронт грузит статические JSON; Go = только пайплайн.
- **Фаза 2 (Go API):** добавляем сервис (`chi`/`Fiber` + Postgres/SQLite) под лидерборд/дейлик/сейвы.
- **Заложить сразу:** доступ к данным за интерфейсом (`DataSource`), чтобы подменить статику на API без переписывания фронта.

### 7.4. Клей — единый контракт данных
JSON Schema как источник истины → генерим TS-типы и валидируем оба конца, чтобы Go-пайплайн и TS-фронт не разъехались.

### 7.5. Итоговая раскладка
```
Go   → ETL-пайплайн (raw → рейтинги → game JSON)     [ядро]
Go   → API (фаза 2: дейлик/лидерборд/сейвы)          [опц.]
TS   → React+Vite фронт + игровая логика             [ядро UX]
Схема → JSON Schema контракт данных                   [клей]
```

| # | Вопрос | Решение |
|---|---|---|
| 1 | Название | ✅ **Aegis Draft** |
| 2 | Runtime MVP | ✅ **static-first** (Go API — фаза 2; это вопрос фич, не масштаба) |

| 5 | Роли | ✅ как в 322-0: `safelane/mid/offlane/support×2`, **без деления 4/5** |
| 6 | Режимы драфта | ✅ **Team Packs** + **Mixed Draft** (5 из разных команд) |

| 11 | Верхнеуровневые режимы | ✅ **Classic / Esports Manager / Real Tournament**; Classic включает Team Packs/Mixed, детали §5.9.1 |

### Осталось обсудить (не блокирует старт)
- **A.** Mixed Draft: строгий порядок ролей `1→5` (дефолт) или разрешить брать любую роль из микс-пака (свободный порядок)?
- **B.** Peak: длина окна (3 или 6 мес.), шаг, точный `N_min`.
- **C.** Точные веса OVR по ролям, коэффициенты сглаживания (`m`, μ), веса team-success (TI/Major/призовые) — подбор на этапе тюнинга.
- **D.** Правовые рамки публичного использования Liquipedia/OpenDota (атрибуция CC-BY-SA обязательна) — насколько важно на старте.
- **E.** Пороги и число этапов полного забега; стартовая гипотеза — 4 этапа + финал.
- **F.** Первый курируемый набор Tactics/Camp actions и лимит слотов; решать через отдельный balance spec и симуляцию.
- **G. Драфт героев из паков — решено и реализовано 2026-07-12.** Забег состоит из 5 пиков игроков + 5 пиков героев. Каждый экран пака предлагает ровно 5 доступных героев, дубли исключены; назначение auto выполняет exact matching, Manual остаётся отдельной задачей T3.10. Инспекция игрока и games player×hero реализованы по UX-паттерну референса без копирования его визуального языка. См. аудит `docs/audits/2026-07-12-hero-draft-stats-chemistry.md`.
- **H. Вердикт итога — решено и реализовано 2026-07-12.** После драфта показываем воспроизводимый прогноз против 18-team field, затем тем же полем играем группы → double-elimination playoffs → Grand Final и выдаём фактическое место. Это не декоративный score verdict: preview и турнир используют один `TournamentEngine`. См. аудит `docs/audits/2026-07-12-tournament-cycle-parity.md` и BACKLOG **TREF2/T5.1**.

## 11. Reference-driven development

Референсы используются как исследовательский материал, а не как безусловная спецификация. Любая просьба сравнить реализацию запускает `reference-parity-audit`: живой проход 322-0/другого референса и Aegis Draft, единый сценарий, матрица `parity / intentional-divergence / defect / missing / not-applicable / unknown`, затем синхронизация подтверждённых выводов с PRD и BACKLOG.

Baseline от 2026-07-11: [reference audit](audits/2026-07-11-322-balatro-baseline.md). Источники продуктовых принципов Balatro: [официальный сайт](https://www.playbalatro.com/) и [официальное описание PlayStation](https://www.playstation.com/en-us/games/balatro/).
