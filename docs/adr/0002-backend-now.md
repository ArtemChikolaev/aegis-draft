# ADR 0002 — Заводим backend сейчас (гибрид: статика + Go API)

- Статус: **Принято** (2026-07-12)
- Контекст: расширенный scope (PRD §5.9 — Esports Manager / Real Tournament / полный roguelite run, лидерборд, сейвы, дейлик). Пересматривает **тайминг** из [ADR 0001](0001-tech-stack.md): «Go API — фаза 2» → **начинаем сейчас**. Static-first для игровых данных сохраняется.

## Решение

**Гибрид, не отказ от static-first:**

| Слой | Технология | Роль |
|---|---|---|
| Игровые данные | **Static JSON на CDN** (как в ADR 0001) | events/packs/players/heroes/… — read-heavy, детерминированные артефакты пайплайна. Не в БД. |
| Пользовательское/общее состояние | **Go API (`server/`, chi) + Postgres** | Аккаунты, сейвы забегов, мета-прогрессия, лидерборд, дейлик-сид, Manager-персистенс. |
| Доступ с фронта | Интерфейс **`DataSource`** (статика) + новый **API-клиент** (динамика) | Разделение: статику не тащим через сервер. |

## Почему так
- Новый scope почти весь **сингловый** и мог бы жить клиентом, но пользователь выбрал ранний backend под будущие кросс-юзер фичи (лидерборд/дейлик/аккаунты/облачные сейвы) и Manager-персистенс — чтобы не переделывать позже.
- **Игровые данные остаются статикой:** это лучшее из ADR 0001 (CDN масштабируется, пайплайн детерминирован). Сервер держит только изменяемое состояние.
- Go на сервере = один язык с пайплайном, переиспользование `internal/model`, `rating`, `scoring` для **серверной ре-симуляции** (анти-чит дейлика/лидерборда).

## Границы / следствия
- `server/internal/{transport, service, store, model}` — слои не пробивать (как в пайплайне). Секреты из env. Миграции версионированы.
- Контракт с фронтом: пересекающиеся сущности — те же `schema/`-формы; динамика (сейвы/лидерборд) — новые эндпоинты.
- **Детерминизм/анти-чит:** дейлик и лидерборд валидируются сервером — счёт/паки должны быть воспроизводимы в Go (переиспользуем rating/scoring из пайплайна), а не доверяться клиенту слепо.
- Хостинг: managed Postgres + один контейнер (Fly.io / Render), **без Kubernetes** (обоснование — ADR 0001 §static-first; k8s не оправдан).

## Решения по стеку (2026-07-12)
- **Auth: Steam OpenID, опционально.** Учитывая 322-0 (нет логина, стата по устройству): по умолчанию **анонимная local-first игра** (сейвы/прогресс в localStorage); Steam OpenID подключается **только** для облачной синхронизации, лидерборда и cross-device. Меньше трения, ближе к референсу.
- **Доступ к БД: `sqlc` + `goose`.** `sqlc` — типобезопасные запросы из обычного SQL (compile-time, без ORM-магии); миграции версионируем `goose` (schema-first). Не `database/sql` (сырой, verbose).
- **Router: `chi`.** Поверх stdlib `net/http` (`http.Handler`), совместим с экосистемой middleware/`context` и остальным Go-кодом. Не Fiber (fasthttp, не `net/http`-совместим, лок-ин; скорость нерелевантна на нашем масштабе).

## Отклонённые альтернативы
- **Статика надолго / только localStorage** — дешевле, но откладывает cross-device, анти-чит и общий лидерборд; выбрали ранний backend.
- **Игровые данные в БД** — избыточно: это детерминированные read-only артефакты, статика дешевле и проще.
- **Kubernetes** — оверинжиниринг для одного сервиса.
