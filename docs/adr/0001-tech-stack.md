# ADR 0001 — Технологический стек и архитектура

- Статус: **Принято** (2026-07-11)
- Контекст: PRD v0.3, §7.

## Решение

| Слой | Технология | Роль |
|---|---|---|
| Data-пайплайн | **Go** | ETL: OpenDota + Liquipedia → игровые JSON. Ядро ценности. |
| Контракт данных | **JSON Schema (draft-07)** | Источник истины (`schema/`), валидация обоих концов. |
| Фронтенд | **TypeScript + React + Vite** | UI + игровая логика счёта на клиенте. |
| State | **Zustand** | Лёгкий стор для состояния забега. |
| Runtime данных | **Static-first** | JSON на CDN; сервер не нужен для базовой игры. |
| API (фаза 2) | **Go** (`chi`/`Fiber`) + Postgres/SQLite | Дейлик, лидерборд, сейвы. |

## Почему так

- **Go для ETL:** конкурентный fetch с rate-limit (goroutines + worker pool), строгая типизация, один бинарник, быстрый JSON. Прозрачная замена «странного парсинга» оригинала.
- **Static-first:** для масштабирования сам по себе лучше и дешевле Go API (CDN масштабируется ~бесконечно). Go API нужен под фичи с общим состоянием, а не под нагрузку → откладываем в фазу 2.
- **JSON Schema как клей:** генерим TS-типы из схемы, валидируем и вход пайплайна, и артефакт → Go и TS не разъезжаются.

## Следствия

- Доступ к данным на фронте прячем за интерфейс `DataSource`, чтобы позже подменить статику на API без переписывания.
- Пайплайн детерминирован и версионирован (`manifest.ratingModelVersion`), пересобирается одной командой.
- Обязательна атрибуция Liquipedia (CC-BY-SA) и корректный User-Agent (иначе бан).

## Отклонённые альтернативы

- **Go API с первого дня** — избыточно для MVP, не решает задач масштаба.
- **Dotabuff как источник** — нет публичного API, скрейпинг против ToS и хрупок; OpenDota покрывает те же данные.
- **Деление саппортов 4/5** — приоритет фарма ненадёжен (PRD §5.1).
