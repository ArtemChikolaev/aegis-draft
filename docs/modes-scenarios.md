# Сценарии верхнеуровневых режимов (BA-спека)

> **Статус:** бизнес-анализ, **без реализации**. Черновик 2026-07-12 по запросу продукта.
> Раскрывает заглушку PRD §5.9.1 («Esports Manager — точные правила: отдельная спека») и
> дожимает Real Tournament из §5.9.1 до играбельного сценария. Перед любым кодом — скилл
> `discovery-before-code` (Правило 0) и `game-state-architecture` (mode shell ≠ RunConfig ≠ RunEngine).
>
> Что уже есть и переиспользуем (доказательство reuse-first ниже в §0) — не изобретаем заново:
> рейтинги игроков, реальные паки-ростеры, `TournamentEngine`, детерминизм по seed, серверные сейвы (ADR 0002).

## 0. Что у нас уже есть в данных (feasibility) — общая база

Прежде чем проектировать экономику/поля, фиксируем, что датасет **реально отдаёт** (OpenDota,
static-first) и что придётся **синтезировать** или отложить (нет Liquipedia-доступа).

| Нужно режиму | Есть в данных? | Источник / примечание |
|---|---|---|
| OVR игрока (event / peak / team-success) | ✅ | `players`, `packs`, rating-модель §5.4 |
| Роли (safelane/mid/off/sup×2) | ✅ | inferred из `lane_role` (S2) |
| Career player×hero (пул героев) | ✅ (дозагружается) | `/players/{id}/heroes`, см. TDATA1 |
| Реальные ростеры команд на событии | ✅ | `packs` (пак = топ-5 состав команды на событии) |
| Chemistry (пожизненные co-games пар) | ✅ | `squadSynergy`/`peers` |
| Успех команды за окно (престиж) | ✅ | `teamSuccess` (1y/2y/5y) |
| **Трансферная стоимость / зарплаты / контракты** | ❌ | **синтезируем** из OVR/престижа/окна (детерминированно) |
| **Возраст / дата рождения игрока** | ❌ | нет в OpenDota; «эра/форма» аппроксимируем осью Format (1y/2y/5y) |
| **Реальные placements / призовые / исход турнира** | ⏸️ deferred | Liquipedia недоступна → поле **симулируем** движком, а не реплеим реальный исход |
| **Регион игрока/команды** | 🟡 частично | из истории команд; для MVP можно опустить |

**Ключевой вывод:** экономика Manager и «стоимость» игроков — **производные от нашей рейтинг-модели**,
а не рыночные данные. Real Tournament воспроизводит **силу** реального поля, но не его исторический
результат (placements отложены). Это честно проговариваем в UI (атрибуция + «симуляция, не запись матча»).

## 1. Esports Manager («Dota Manager»)

### 1.1. Питч / фантазия игрока
Ты — не фэнтези-драфтер на один вечер, а **менеджер организации**: с бюджетом, а не всеведением.
Нельзя просто взять пятёрку топ-OVR — на неё не хватит денег. Собираешь состав по средствам, выигрываешь
турниры, на призовые усиливаешь ростер и **лезешь вверх по дивизионам**, поднимая рейтинг команды.
Ценность игрока — не только OVR, но и **цена, роль, окно формы, риск**.

### 1.2. Ядровой цикл
```
Старт: организация + стартовый бюджет + пустой ростер
  → ТРАНСФЕР-ОКНО: купить/продать игроков в рамках бюджета (+ зарплатный потолок)
  → ТУРНИР: сыграть (переиспользуем TournamentEngine) против поля своего дивизиона
  → ПРИЗОВЫЕ: деньги за место + бонус за overperformance
  → МЕТА-РЕШЕНИЕ: усилить сейчас vs накопить на следующий дивизион
  → повторить; цель — подняться из открытых квалов в tier-1 и поднять Team Rating
Провал: бюджет < обязательных расходов (зарплаты) → релегация / конец сейва
```
Это **длинный сейв** (в отличие от одноразового Classic-забега). По ADR 0002 состояние сейва
(бюджет/ростер/дивизион/сезон) держит **сервер** (`server/internal/{store,model}`), игровые данные — static-first.

### 1.3. Экономика (всё детерминировано по seed + версия данных)
- **Цена игрока** = `f(OVR, престиж команд за окно, свежесть окна, дефицит роли)`. Синтезируем формулой
  (конкретные веса — отдельный balance-spec, версионируется как `economyModelVersion`). Пример логики:
  звезда чемпионского состава last_1y дороже того же OVR из команды-аутсайдера или из «пыльного» last_5y.
- **Зарплаты/контракты** (💡 то, что делает режим менеджером, а не Classic-скином): у игрока есть
  зарплата, списываемая каждый этап, и срок контракта. Дорогая суперзвезда истощает бюджет → приходится
  балансировать « 1 звезда + 4 роляльщика » vs «5 ровных». Бенч тоже стоит денег.
- **Призовые**: сумма за место в турнире (шкала зависит от дивизиона) + бонус, если прыгнул выше прогноза.
- **Трансфер туда-обратно**: продать игрока, вернув часть стоимости (с дисконтом) — источник ликвидности.

### 1.4. Прогрессия
- **Дивизионы/лестница**: открытые квалы → tier-2 → tier-1 (маппим на реальные слои сцены из наших событий).
  Вход в дивизион выше требует Team Rating / денег; призовые и цены игроков растут с дивизионом.
- **Team Rating** команды растёт от результатов (не «вечный +OVR аккаунту» — это осознанно запрещено §5.10).
- **Сезоны**: несколько турниров = сезон; между сезонами — трансфер-окно и, возможно, инфляция цен.

### 1.5. 💡 Идеи-улучшения (маркируем как предложения, не решения)
1. **Динамика рынка**: цена игрока растёт после сильных выступлений (у тебя или глобально в датасете) и
   падает в простое → механика «купи-на-дне / продай-на-пике», buy-low талантов.
2. **Скаутинг + скрытый рейтинг**: в низких дивизионах/регионах игроки дешевле, но их точный OVR скрыт;
   scouting-действие (из Camp §5.10) раскрывает — риск/награда разведки.
3. **Форма/эра через ось Format**: покупка «легенды на last_5y» — дёшево, но форма угасающая; «звезда last_1y» —
   дорого и на пике. Одна ось Format закрывает и «возраст», которого нет в данных.
4. **Банкротство как fail-state**: бюджет < зарплат → релегация или конец сейва (создаёт напряжение решений).
5. **Синергия дороже суммы**: Chemistry/Hero Synergy дают Team OVR больше, чем сумма OVR, — стимул покупать
   реально сыгранные пары (у нас это есть в `squadSynergy`), а не набор дорогих одиночек.

### 1.6. Переиспользование / границы
- **TournamentEngine** — тот же (поле дивизиона вместо фэнтези-ботов; сила = OVR купленного ростера).
- **Rating/packs** — цена и сила берутся из существующей модели, не новая формула Team OVR.
- **Server (ADR 0002)** — сейв менеджера в `store`; ре-симуляция/валидация на Go (переиспуя `pipeline/internal/{model,rating}`), клиент не источник истины по деньгам.
- **Mode shell**: `selectedMode = "manager"` ≠ `RunConfig` (исполнимые опции сейва) ≠ `RunEngine`/движок экономики (чистая логика).

### 1.7. Открытые BA-решения (нужно согласовать до кода)
- **M-A.** Длина сейва: бесконечная лестница vs фиксированные N сезонов с концовкой?
- **M-B.** Пул игроков: все ~1300 pack-игроков сразу, или только текущий дивизион/эра?
- **M-C.** Зарплаты — обязательная механика MVP или пост-MVP (сначала только цена покупки)?
- **M-D.** Онлайн-сейв (сервер) сразу или локальный сейв на MVP, сервер позже?

### 1.8. MVP-срез (минимум, доказывающий петлю)
Стартовый бюджет → купить 5 по ролям в рамках денег (цена = функция OVR/окна) → 1 турнир (TournamentEngine) →
призовые → 1 трансфер-окно (купить/заменить) → повтор. Без зарплат/скаутинга/маркет-динамики — их вводим по телеметрии.

## 2. Real Tournament

### 2.1. Питч / фантазия игрока
Не выдуманные боты, а **настоящий турнир с настоящим полем**: берётся реальный ивент, и все команды в нём —
это их **реальные ростеры целиком** (напр. Falcons = все 5 реальных игроков Falcons на том событии). Ты —
**команда-челленджер**: собираешь пятёрку из тех, кого в этом турнире нет — легенд прошлого, ветеранов,
звёзд других эпох/сцен — и проверяешь, обыграет ли твой «dream team» реальное поле.

### 2.2. Сценарий
```
Seed выбирает реальный tournament snapshot (из наших tier-1 событий)
  → показываем поле: 16–20 реальных команд с их полными ростерами (roster lock)
  → все залоченные accountId ИСКЛЮЧАЮТСЯ из твоего draft pool
  → собираешь challenger-пятёрку по ролям из оставшихся (легенды/ветераны/др. эпохи)
  → играешь то же поле через группы → плей-офф → финал (TournamentEngine)
  → итог: твоё место среди реального поля
```

### 2.3. Ключевой инсайт переиспользования
Это **не новый движок**, а `TournamentEngine` с двумя подменами:
- `opponentPool` = **реальные `packs` выбранного события** (у нас уже пак = топ-5 реальный состав команды),
  вместо фэнтезийных ботов Classic;
- **фильтр draft pool**: eligibility по каноническому `accountId` (PRD §5.9.1) — залоченные игроки поля
  вычёркиваются из выбора пользователя.
То есть сила соперников — честная (из реальных OVR), а не рандом. Архитектурно почти всё готово.

### 2.4. Честность и кросс-эра рейтинг
- **Реальный исход не реплеим**: placements/призовые отложены (нет Liquipedia) → мы воспроизводим **силу**
  поля и **симулируем** по seed, а не показываем, кто реально выиграл. В UI: «симуляция против реального
  поля», без ложного «так и было». (Если позже появятся placements — можно показывать реальный результат рядом.)
- **Легенда оценивается формой своей эпохи**, а не сегодня (PRD §5.9.1): challenger из 2015 берёт свой
  peak/оконный OVR той эпохи. 💡 Открытый вопрос — нормализация OVR между эпохами (мета/инфляция рейтингов):
  играть «как было тогда» честнее, но может быть несбалансированно против свежего поля.
- **Инвариант lock**: если после roster lock нельзя собрать валидную пятёрку ролей — генератор меняет
  snapshot/pool или падает понятной ошибкой, но **молча lock не ослабляет** (уже в PRD §5.9.1).

### 2.5. 💡 Идеи-улучшения
1. **Underdog-seed**: показать твой прогноз-место среди реального поля до старта (реюз `projection`), чтобы
   был явный вызов «ты 18-й из 18, пробей выше».
2. **Тематические челленджи**: «собери состав только из TI-чемпионов прошлого», «только СНГ-ветераны» —
   курируемые фильтры пула поверх базового режима (переиспуют ту же eligibility-механику).
3. **Замена одной реальной команды** (альт-вариант, пост-MVP): вместо сбора челленджера — «прими одну
   реальную команду поля и доведи её выше её реального сида». Другой вкус, тот же движок.

### 2.6. Данные / переиспользование / открытые решения
- **Данные:** всё есть (`events`, `packs`, `players`, rating). Deferred — только реальные placements (для «как было»).
- **Переиспользование:** `TournamentEngine`, `packs`, eligibility по `accountId`, `projection`.
- **Открытые BA-решения:**
  - **RT-A.** Выбор snapshot: только по seed (roguelite) или дать список реальных ивентов на выбор?
  - **RT-B.** Кросс-эра рейтинг: «форма эпохи» как есть vs нормализация к полю снапшота?
  - **RT-C.** Размер challenger-пула: все вне-турнирные игроки всех эпох, или ограничить окном/престижем?

### 2.7. MVP-срез
Seed → 1 реальное событие → поле = его реальные паки (roster lock) → собрать challenger-5 из оставшихся →
турнир через `TournamentEngine` → место среди реального поля. Без тематических фильтров/альт-варианта.
